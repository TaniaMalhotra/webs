That problem can be solved (at least in modern browsers) by using the pattern attribute.
For instance, if you expect an integer value as the input, use:
<input type="text" inputmode="decimal" pattern="\d+"/>
Entering “n.a.” in such a field would cause the browser to highlight the field (usually in red) and display an error message.

]]> By: Adrian Roselli https://css-tricks.com/better-form-inputs-for-better-mobile-user-experiences/#comment-1755973 Wed, 22 Apr 2020 00:15:30 +0000 https://css-tricks.com/?p=306387#comment-1755973 To supplement the author’s great info…

Using autocomplete is not only a potential usability boon, as you outline, it also currently the only method to satisfy Success Criterion 1.3.5 Identify Input Purpose (Level AA) from WCAG 2.1. There is a slightly different list of autocomplete values in WCAG 2.1, which are intended to relate to the user of the content and pertain only to information related to that individual.

Even as WHATWG adds more autocomplete values, the WCAG values are unlikely to change immediately. So bear that in mind if you are reading this a year or two down the road and trying for the UX and WCAG win all in one.
